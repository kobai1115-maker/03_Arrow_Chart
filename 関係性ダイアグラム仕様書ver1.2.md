# Arrow Chart & Care Plan AI App Specification

## 1. Project Overview

医療・介護現場でのアセスメントと思考過程を可視化する「関係性ダイアグラム」を作成し、その構造から「短期目標」と「長期目標」の候補を自動抽出するマルチプラットフォーム対応（Web, iOS, Android, iPadOS）のアプリケーションを開発する。
インターライ（interRAI）のCAPsの思考過程をベースとし、論理破綻を防ぐ厳密な結線ルールをシステムに組み込む。

## 2. Tech Stack

* **Framework:** Flutter (Dart)
* **Diagram Engine:** `diagram_editor` パッケージをベースに拡張、または `CustomPainter` を用いた独自実装
* **Architecture:** MVVM または Riverpod を用いた状態管理
* **Local Storage:** SQLite または shared_preferences (オフラインファースト設計：常時ネットワーク接続を不要とする)

## 3. Arrow Chart Logical Structure & Validation

ユーザーが配置するノードとエッジには厳密な論理的意味を持たせる。

### 3.1 Nodes (要素)

1. **Subjective (主観的事実):**
* 形状: 四角形（□）
* 意味: ユーザーや家族の訴え、感情、希望（例：「〜したい」）。


2. **Objective (客観的事実):**
* 形状: 円形（○）
* 意味: 診断名、身体状況、環境などの事実（例：歩行不安定、下肢筋力低下）。



### 3.2 Edges (関係性)

1. **Cause & Effect (順接):**
* 線種: 通常の矢印（→）
* 意味: 原因と結果の因果関係（〜なので〜である）。


2. **Paradox (逆説):**
* 線種: 中心にギザギザ波形がある直線（─ギザギザ─）※CustomPainter等で描画
* 意味: ギャップや相反する状態（〜だけど〜である）。ニーズの構造を形成する。


3. **Meaning / Background (意味づけ・背景):**
* 線種: 直線（ー）
* 意味: 対象者の価値観や背景。順接だが直接的な因果関係とは切り離す。
* 例: [□: 栄養バランスよく食事をとりたい] ー [□: 食事は健康維持に大切なこと]



### 3.3 Connection Validation Rules (重要制約)

* **一方向性の担保:** ノード①からノード②へ「順接(→)」「逆説」「意味づけ」がある場合、②から①へは「順接」「逆説」「意味づけ」のいずれの線も引けない。
* **双方向の抑止:** ①⇔②のように双方向に順接が向かう状態（情報不足・論理破綻）はシステムエラーとしてブロックする。必ず間に別のノードを挟んで円環を作るようユーザーに促す。

## 4. Core UI/UX Requirements

* **Touch-First:** モバイル対応のため、タップやドラッグのヒット領域は最低 48x48dp を確保する。
* **Tap-to-Connect:** ドラッグでの結線は誤操作を招くため、「ソースをタップ → ターゲットをタップ」の順次タップで結線できるモードを用意する。
* **Midpoint Connection (中点からの派生結線):**
* 「意味づけの直線（ー）」および「逆説のギザギザ線」をタップした際、線の中央に仮想の「接続ハンドル」を出現させる。
* ユーザーはこの中点ハンドルをソースとしてタップし、別の新規ノードへ結線（派生）できること。



## 5. Goal Extraction Algorithm (AI Suggestion Logic)

キャンバスのグラフ構造（JSON）を解析し、以下の厳密なルールで目標候補を抽出後、LLM APIに渡して自然な文章に変換して提示する。

1. **ニーズの抽出:**
* 原因分析の到達点である「○（客観）」と、それに「逆説（ギザギザ線）」で繋がる「□（主観）」のペアを検索。
* この「□（主観）」の内容をニーズ（〜したい）とする。


2. **長期目標の抽出:**
* 上記で見つかった「○（客観）」（結果の到達点）を長期目標のベースとし、「〜できる」というポジティブな表現に変換する。


3. **短期目標の抽出:**
* 結果の「○（客観）」から、「順接（→）」を逆方向に遡る。
* 最も上流にある「○（原因となる客観的事実）」を短期目標とする。
* *例外処理:* 最上流が「転倒」「骨折」など解消不可能な事実の場合、1つ下流に戻り、解決可能な事実を短期目標として抽出する。


4. **悪循環（ループ）の解釈:**
* 遡る過程で順接がループしている場合は「悪循環」と解釈し、そのループ群の中で最も介入が容易な事実を短期目標の候補としてAIに推論させる。

5. **特殊記号の解釈:**
* ノード内のテキストに特定の記号（「!」「!!」「?」または全角の「！」「！！」「？」）が含まれる場合、AIがそれぞれ「将来予測」「反復」「仮説」として解釈し、ケア指針の生成においてそれらに対する言及や分析を含めるようにする。



## 6. interRAI CAPs Integration

* グラフ上の「○（客観的事実）」のテキスト内容をAIに解析させ、interRAIのCAPs領域（機能的パフォーマンス、認知など）にマッピングする。
* 「機能低下の予防」か「改善の促進」の2段階のリスク評価を行い、エビデンスに基づいたケア指針（ガイドライン）を第2表の提案として出力する。

## 7. Business Logic Notes

* アプリはサブスクリプション制を前提とするため、AI（LLM）呼び出し部分には将来的に回数制限（レートリミット）やAPIキーの管理ロジックを挿入できるよう、Service層を疎結合にしておくこと。

## 8. Offline & Online Specifications

本アプリケーションは、訪問介護・看護など電波の不安定な現場での利用を想定し、**オフラインファーストアーキテクチャ**を採用する。

### 8.1 Offline Capabilities (完全オフライン対応機能)
以下の機能はネットワーク接続なしで動作しなければならない。
* **チャートの作成・編集**: ノードの配置、エッジの結線、テキスト入力などキャンバス上のすべての操作。
* **データの永続化**: SQLite (`drift` パッケージ等を利用) への Diagram および Node/Edge データのローカル保存・読み込み。
* **エクスポート機能**: 端末のネイティブAPIを利用したPDFおよび画像（PNG）の生成と保存・共有機能（`printing`, `share_plus` 等の利用）。

### 8.2 Online Capabilities (オンライン必須機能)
以下の機能はインターネット接続が必要であり、オフライン時には適切なエラーハンドリング（ユーザーへの通知）を行うこと。
* **AI Care Plan Suggestion**: Gemini API を呼び出して実行する「ニーズ・目標・ケア指針の生成（`LlmService` クラスのメソッド呼び出し）」。
* オフライン時に「✨（AI提案）」ボタンが押下された場合は、「インターネット接続が必要です。作成したチャートは保存されているので、通信環境の良い場所で再度お試しください。」等のスナックバー／ダイアログを表示するロジックを実装する。
