# マルチプラットフォーム環境における関係性ダイアグラム描画アプリケーション仕様策定に関する総合研究報告書

## 1. 序論およびプロジェクトのコンテキスト

論理的なフロー、プロセスの順序、および因果関係を視覚化する関係性ダイアグラム（Arrow Chart）は、多岐にわたる専門分野で不可欠なツールとして機能している。とりわけ、日本の医療・介護分野においては、包括的アセスメント手法であるインターライ（interRAI）のCAPs（Clinical Assessment Protocols：臨床アセスメントプロトコル、現在はCollaborative Action Plans：協働的ケア計画とも呼ばれる）の思考過程を源流とし、ケアプランの作成やアセスメントを可視化するための標準的な手法として定着しており、複雑な情報を体系的に整理・共有するための基盤となっている。これらの業務は従来のオフィス環境におけるデスクトップPC上での作業に留まらず、訪問先でのタブレット（iPad、Android）操作や、移動中のスマートフォン操作など、多様なフィールド環境で実行されることが大前提となっている。

本報告書は、Webブラウザ、スマートフォン（iOS/Android）、およびタブレット（iPad/Android）上でシームレスかつ高性能に動作する関係性ダイアグラム描画アプリケーションの開発に向けた、包括的な仕様策定のための調査結果を論じたものである。クロスプラットフォーム対応のアプリケーションを構築する上で、高度なグラフィック描画性能の確保、プラットフォーム間でのコード再利用性の最大化、そしてマウス操作とは根本的に異なるタッチデバイスに最適化されたユーザーエクスペリエンス（UX）の確立が、本プロジェクトの成否を分ける中核的な要素となる。本調査では、フロントエンドアーキテクチャの選定から、描画エンジンの比較、モバイルおよびタブレット特有のインタラクション設計、そしてデータフローやパフォーマンス最適化に至るまで、多角的な視点から仕様策定に必要なインサイトを提供する。

## 2. マルチプラットフォーム開発アーキテクチャの比較と選定

マルチプラットフォーム対応のアプリケーションを構築する上で、基盤となるフレームワークの選定は、開発コスト、将来の保守性、および描画パフォーマンスに決定的な影響を与える。現在、モバイルおよびWeb開発の主流となっている3つのアプローチ、すなわちProgressive Web Apps（PWA）、React Native、およびFlutterについての技術的メカニズムと、関係性ダイアグラムという高度なグラフィック処理を伴う要件に対する適合性を評価する。

Progressive Web Apps（PWA）は、既存のWeb開発のスキルセットを活用し、単一のWebアプリケーションを各デバイスのブラウザ上でネイティブアプリのように動作させるアプローチである。この手法の最大の優位性は、開発およびデプロイが最も迅速であり、App StoreやGoogle Playなどのアプリストアの厳格な審査プロセスを迂回できる点にある。しかしながら、PWAはブラウザのサンドボックス環境内で動作するという構造的な制約を抱えている。そのため、デバイス固有のハードウェア機能やネイティブAPIへのアクセスに限界があり、カメラやGPSといった一部の機能を除き、ネイティブアプリと同等の深いシステム連携を実現することは困難である。さらに、描画処理がDocument Object Model（DOM）を介して行われるため、数百のノードと複雑なパス（矢印）を持つ関係性ダイアグラムをタッチ操作でパン・ズームする際、ネイティブアプリに比べて応答速度の遅延やフレームレートの低下（Jankの発生）が生じるリスクが極めて高い。

次に、Meta（旧Facebook）によって開発されたReact Nativeは、JavaScriptまたはTypeScriptを用いてiOSおよびAndroidのネイティブコンポーネントを駆動するフレームワークである。React Nativeの主要な利点は、iOSとAndroid間で最大95%のコードを共有できる点と、世界最大規模であるReactの巨大なエコシステムおよびサードパーティライブラリをそのまま活用できる点にある。Web開発（React.js）の深い知識を持つ開発チームであれば、学習コストを最小限に抑えつつ迅速に開発を進めることが可能である。しかし、React Nativeにはアーキテクチャ上の致命的なボトルネックが存在する。それは、JavaScriptを実行するスレッドと、デバイスのネイティブUIをレンダリングするスレッド間の通信を担う「ブリッジ」のオーバーヘッドである。標準的なフォーム入力やリスト表示といったUIの構築には全く問題ないが、関係性ダイアグラムのような高頻度で座標計算と再描画（ドラッグ＆ドロップ、アニメーション、リアルタイムのパス再計算）が連続して求められるグラフィックス集約型の処理においては、この非同期のブリッジ通信がパフォーマンスの顕著な低下を招く懸念がある。

最後に、Googleが主導するFlutterは、Dart言語を使用し、単一のコードベースからiOS、Android、Web、さらにはmacOSやWindowsなどのデスクトップ向けにネイティブコンパイルされたアプリケーションを構築するUIツールキットである。Flutterの最大の特徴は、実行時にOSが提供するOEMウィジェット（ネイティブUIコンポーネント）を一切使用せず、独自のレンダリングエンジン（歴史的にはSkia、現在は最新のImpellerエンジン）を用いて画面のすべてのピクセルを直接描画する点にある。このアプローチにより、React Nativeが抱えるブリッジ通信のオーバーヘッドを完全に排除し、モバイルデバイス上で60fps、あるいは対応デバイスでは120fpsの極めて滑らかなパフォーマンスを安定して実現する。さらに、開発段階においては「ホットリロード（Hot Reload）」機能により、状態を保持したままコードの変更を即座にUIに反映させることが可能であり、UI/UXのイテレーションを劇的に加速させる。Dart言語の学習という初期コストは発生するものの、一度習得すれば、ピクセル単位での完全なカスタマイズが可能であり、複雑なグラフィックエディタの構築に極めて適したアーキテクチャであると言える。

| 評価基準 | Progressive Web Apps (PWA) | React Native | Flutter |
| --- | --- | --- | --- |
| **描画パフォーマンス** | 中（ブラウザのDOMレンダリング性能に依存） | 高（ただし非同期ブリッジ通信の制約あり） | 最高（Skia/Impellerによる直接レンダリング） |
| **プラットフォーム間のコード共有率** | 100%（完全に同一のWebコード） | 約90〜95%（一部ネイティブモジュールが必要） | 約95〜100%（UIも完全に共有） |
| **ネイティブAPIへのアクセス** | 限定的（Geolocationやカメラ等一部機能のみ） | 拡張可能（サードパーティ製ネイティブモジュール） | 完全アクセス可能（Platform Channels等を利用） |
| **UIのカスタマイズ性と柔軟性** | 高（CSSによる自由なスタイリング） | 中（プラットフォーム固有の制約を受ける） | 最高（ピクセル単位の完全な制御が可能） |
| **オフライン・ローカルデータ対応** | Service WorkerとIndexedDBによるキャッシュ | AsyncStorageやRealm等による非同期データ管理 | SQLite等のローカルDBとの完全かつ同期的な統合 |

これらのアーキテクチャを比較分析した結果、導き出される技術的推論は明確である。関係性ダイアグラムの描画という「無限キャンバス上で複数のノードを動的に結線し、複雑なパン・ズーム操作をリアルタイムで行う」ユースケースにおいては、DOM操作の限界に縛られない描画エンジンが絶対的に必須となる。特に、建設現場や医療・介護現場など、オフライン環境下で手袋を着用しながらの操作も想定される過酷なフィールド環境においては、タッチ応答性に優れ、ネイティブと同等かそれ以上の描画速度と安定性を担保できるFlutterが、本プロジェクトの要件を満たす最も合理的な選択肢となる。React Nativeは既存のJavaScript資産を活かす場合には強力な選択肢となるが、グラフィックス集約型のインタラクションにおいては、Flutterの直接レンダリング方式がもたらす恩恵がそれを上回る。

## 3. 関係性ダイアグラム描画エンジンの技術的パラダイムとライブラリ評価

選択するアーキテクチャ（Web/Reactベース、あるいはFlutterベース）に応じて、採用すべきダイアグラムライブラリのパラダイムは大きく異なる。複雑なノード管理、パスルーティング、およびキャンバスの仮想化をゼロから独自開発することは膨大な工数とリスクを伴うため、既存の成熟したライブラリのコアエンジンを活用し、要件に合わせて拡張するアプローチが標準的である。本節では、データ可視化ライブラリとノードベースのダイアグラムライブラリの違いを明確にした上で、主要なライブラリを評価する。

まず重要な前提として、「データの可視化（Data Visualization）」と「ダイアグラムの描画（Diagramming）」は似て非なる技術領域であることを認識する必要がある。市場にはVictory Native、React Native SVG Charts、ECharts、Syncfusion Charts、Graphic、あるいは最新のCristalyseなど、極めて優秀なチャートライブラリが存在する。これらは棒グラフ、円グラフ、散布図などを美しく描画し、大量のデータを集計して提示する用途（ダッシュボード等）には最適である。しかし、関係性ダイアグラムが要求するのは、ユーザー自身が任意の場所にノードを配置し、それらを論理的なパス（矢印）で動的に接続し、さらにその構造を編集できる「ノードベースのエディタ」としての機能である。したがって、単なるデータのプロットツールではなく、イベントハンドリング、衝突判定、パスルーティングのアルゴリズムを内包したダイアグラム専用のライブラリを選定しなければならない。

### 3.1 JavaScript / Reactベースのダイアグラムライブラリ

Web技術（PWAまたはReact Native Web）を基盤とする場合、以下のライブラリが主要な候補として挙げられる。これらのライブラリはDOM、SVG、またはHTML Canvasを用いてレンダリングを行う。

#### React Flow

Reactエコシステムに特化して設計された、オープンソースのノードベースダイアグラムライブラリである。React Flowの最大の特徴は、ReactのHooks（`useNodesState`, `useEdgesState`等）と完全に統合されており、React開発者にとって極めて直感的なDX（Developer Experience）を提供する点である。さらに、タッチデバイスへのネイティブ対応が強力であり、ピンチジェスチャーによるキャンバスのズームや、ドラッグによるパンが標準で組み込まれている。また、モバイル環境におけるノード結線において、小さな画面でのドラッグ操作が困難な状況を解決するため、タップによる順次選択（ソースのハンドルをタップし、次にターゲットのハンドルをタップして結線する仕様）をサポートしており、優れたモバイルUXを容易に実装できる。

#### JointJS vs GoJS

商用エンタープライズ要件で広く用いられる成熟したJavaScriptダイアグラムライブラリであり、両者はレンダリングのアプローチにおいて対極に位置する。
JointJSは、SVG（Scalable Vector Graphics）を使用してレンダリングを行う。ノードやリンクがDOM要素として描画されるため、CSSによる柔軟なスタイリングや、個別の要素に対する詳細なイベントハンドラ（クリック、タッチ、ホバー）の付与が極めて容易であり、インタラクティビティに優れるという特性を持つ。SVGは要素数が増えるとDOMの負荷が高まるという弱点があるが、JointJSは非同期の仮想レンダリングアルゴリズムを実装しており、ビューポート外の要素をカリングすることで、大規模なダイアグラムでもパフォーマンスを維持する機能を有している。
一方、GoJSは主にHTML Canvas要素を使用してレンダリングを行う（ビットマップベースの描画）。Canvasアプローチの最大の利点は、ブラウザのDOMツリーを肥大化させないため、数万のノードを持つ巨大で複雑なネットワーク図やSCADAシステムのようなデータ集約型環境において、圧倒的なパフォーマンスとスケーラビリティを発揮する点である。しかし、個々のノードがDOM要素ではないため、Reactコンポーネントをノード内に直接埋め込んだり、標準的なCSSでスタイリングしたりすることが難しく、Reactエコシステムとの統合には専用のラッパーコンポーネントを用いた慎重な実装が要求される。

### 3.2 Flutterベースのダイアグラムライブラリ

前節でのアーキテクチャ評価に基づき、Flutterを基盤として採用した場合、以下のパッケージが有力な描画エンジンとなる。FlutterはすべてのUIを「ウィジェット（Widget）」というツリー構造で管理し、それをSkiaやImpellerエンジンで直接描画するため、SVGとCanvasの利点（高いカスタマイズ性と優れたパフォーマンス）を併せ持つ環境が提供される。

#### `diagram_editor`

Flutter環境においてカスタムダイアグラムエディタを構築するために設計された強力なオープンソースパッケージである。このパッケージは`DiagramEditor`ウィジェットを提供し、カスタムコンポーネント（ノード）の描画、ドラッグ＆ドロップによる自由な移動、キャンバスのシームレスなズームおよびパン操作といった、エディタに必要な基盤機能の大部分をカプセル化している。最大の利点は、デザインおよび振る舞いの完全なカスタマイズが可能である点である。関係性ダイアグラム特有の「四角形（主観的事実）」や「円形（客観的事実）」といった特定の形状や、カスタムノード内のテキスト編集フィールドなどを、標準的なFlutterウィジェットとして自由に定義し組み込むことができる。また、「逆説」を示すギザギザ波形の接続線を描画する場合、Flutterの`CustomPainter`を利用して線定義パラメーターを受け取り、始点と終点の間に波形パスを動的に生成するカスタムリンクのロジックを実装することで対応可能である。

| ライブラリ名 | 対応環境 | レンダリング方式 | 主な特徴と強み | タッチ・モバイル対応 |
| --- | --- | --- | --- | --- |
| **React Flow** | Web / React | DOM (HTML/SVG) | React Hooksと完全統合。カスタマイズ性が極めて高い。 | 優秀（タップ結線対応） |
| **JointJS** | Web (Vanilla/React等) | SVG | 豊富なプラグイン、BPMNサポート、強力なインタラクティビティ。 | 良好（マルチタッチ対応） |
| **GoJS** | Web (Vanilla/React等) | HTML Canvas | 大規模データ処理における圧倒的なスケーラビリティ。 | 良好 |
| **diagram_editor** | Flutter | Flutter Widget/Skia | 高度なカスタマイズ性。あらゆるFlutterウィジェットをノード・リンク化可能。 | 優秀（ネイティブジェスチャー） |

本仕様策定におけるライブラリ選定のインサイトとして、関係性ダイアグラムは単なる静的な可視化図ではなく、ユーザーが直接ノードを操作し、経路（アロー）を動的に引き直し、プロパティをリアルタイムで編集する「高度な編集ツール」としての性質を持つことが挙げられる。高いカスタマイズ性とクロスプラットフォームでのネイティブパフォーマンスを両立させるため、本仕様においては**Flutterアーキテクチャの採用を前提とし、`diagram_editor`パッケージをコアエンジンとして採用しつつ、要件に応じてカスタムのパスルーティング（波形・矢印描画）アルゴリズムを実装するアプローチを推奨する**。

## 4. モバイルおよびタブレット環境におけるUX/UI設計のベストプラクティス

WebブラウザやデスクトップPC上でマウス操作を前提として設計されたダイアグラムエディタを、スマートフォンやタブレットといったタッチデバイスにそのまま移植することは、ユーザビリティの深刻な欠如を招く。モバイル環境では、「ホバー（マウスオーバー）状態の欠如」「極端に限られた画面サイズ」「指（Fat Finger）によるタップ精度の低下」という3つの物理的制約を克服するユーザーエクスペリエンス（UX）設計が不可欠である。

### 4.1 タッチファースト（Touch-First）のインタラクション設計

ユーザーインターフェースは、後付けではなく初期段階からタッチ操作を最優先（Touch-first）として設計されなければならない。

* **タッチターゲットの物理的最適化:** 人間の指による操作エラー（誤タップ）を防ぐため、すべてのインタラクティブな要素（ノード、接続ポイントであるハンドル、ツールバーのアイコンなど）は、十分な物理的サイズを確保する必要がある。AppleのHuman Interface Guidelines（HIG）は最小44x44ptを推奨しており、AndroidのMaterial Designは最小48x48dpを厳格に推奨している。
* **ノード結線UXのパラダイムシフト（Tap-to-Connect）:** デスクトップ環境では、あるノードの端から別のノードへとマウスをドラッグして矢印を引く操作が一般的である。しかし、HTMLのDrag and Drop APIはタッチデバイスでは適切に動作しないことが多く、画面の小さなスマートフォン上でノード間に指を引きずって結線する操作は、キャンバス自体のパン操作（スクロール移動）と競合しやすく、ユーザーに極めて高いフラストレーションを与える。この問題に対するベストプラクティスは、**「ソースノードのハンドルをタップし、次にターゲットノードのハンドルをタップする」という順次タップ方式（Tap-to-Connect）**を採用することである。

### 4.2 ジェスチャーマッピングとビューポート制御

関係性ダイアグラムエディタにおける基本的な操作体系は、iOSおよびAndroidのOS標準ジェスチャーに準拠し、ユーザーの直感に逆らわないように適切にマッピングされなければならない。キャンバスの制御は、デジタル地図アプリ等で用いられる「Slippy Maps」の挙動にインスピレーションを得た操作感が理想的である。

**コンテキスト連動型ズーム（Contextual Zoom）の導入:**
関係性ダイアグラムが複雑化し、数十の要素がキャンバス上に配置されると、モバイルの狭い画面ではノード内の詳細なテキスト（アセスメント内容など）が物理的に読み取れなくなる。この認知負荷の問題に対処するためには、「Contextual Zoom」メカニズムの実装が極めて有効である。ズームアウト状態ではノードの形状（□や○）と矢印のフローのみを表示し、ズームインした状態に達した時点ではじめて内部の詳細テキストを描画するよう動的に切り替える。

## 5. iPadおよびAndroidタブレット特有の高度なインタラクション設計

タブレットデバイスは単なる「画面の大きなスマートフォン」として扱ってはならない。特にiPadOSにおいては、広大な画面領域と、物理キーボード、トラックパッド、そしてApple Pencilという多様な入力デバイスを統合したハイブリッドなエコシステムが形成されている。

### 5.1 画面の有効活用とモダリティ（Modality）管理

タブレットの広大なディスプレイを最大限に活用するため、画面遷移においてモーダルウィンドウは極力避け、ユーザーが元のキャンバスのコンテキスト（全体図）を常に視覚的に維持できるように設計する。ノードのプロパティを編集する際は、ノンモーダルなシート（Bottom SheetやSide Panel）を使用する。

### 5.2 スタイラスペン（Apple Pencil等）の深い統合

* **ホバー（Hover）フィードバック:** Apple Pencilが画面に触れる前に数ミリ浮かせた状態を検知し、接続可能なノードのハンドルを強調表示するなどのプロアクティブなフィードバックを提供する。
* **ダブルタップ・ジェスチャー:** Pencilの側面を指でダブルタップした際の動作（例：ノード追加モードと結線モードの切り替え）をサポートする。

## 10. ケアプラン（短期・長期目標）AI提案機能の論理構造要件

医療・介護現場のアセスメントにおいて、作成された関係性ダイアグラムから「ニーズの構造」をプログラムおよびAIによって自律的に解釈し、「短期目標候補」および「長期目標候補」を抽出・表示する機能の仕様を定義する。
本機能は、LLMの自由な推論に頼るのではなく、チャート上に表現された**厳密なグラフ理論と論理関係のルール**に基づいて候補を特定する。

### 10.1 ノードおよびエッジの論理的定義

ユーザーがキャンバスに描画する要素は、以下の論理的意味を持つデータとして扱う。

*   **ノード（情報の実体）:**
    *   **□（四角形）:** 主観的事実（ユーザーや家族の訴え、感情、希望など）
    *   **○（円形）:** 客観的事実（診断名、身体状況、環境、観察事実など）
*   **エッジ（情報の関係性）:**
    *   **→（矢印）:** 順接（〜なので〜である）。原因と結果の関係（原因分析）を示す。
    *   **─（中心にギザギザ波形の線）:** 逆説（〜だけど〜である）。相反する状態や、解決すべきギャップを示す。
    *   **─（直線）:** 単なる接続や関連。

### 10.2 ニーズの構造検出と目標抽出アルゴリズム

アプリケーション内のロジック（ViewModelまたは専用の解析Service）は、保存されたグラフデータから以下の条件に合致するパスを探索し、目標の抽出を行う。

1.  **ニーズの抽出（ニーズの構造の特定）:**
    *   **条件:** 原因と結果の連鎖（原因分析）の最後の到達点である「○（客観的事実）」に対し、「□（主観的事実）」が「─（ギザギザの逆説エッジ）」で結ばれている関係性を検索する。
    *   **抽出ルール:** このギャップ状態が「ニーズの構造」であり、そこに位置する「□（主観的事実）」（基本的に「～したい」と表現される内容）を「ニーズ」の候補として抽出する。
2.  **長期目標の抽出:**
    *   **条件:** 上記の「ニーズの構造」において、ニーズ（□主観的事実）と逆説で直接結ばれている「○（客観的事実）」（結果の最後の到達点）を対象とする。
    *   **抽出ルール:** この「○（客観的事実）」を長期目標の候補として抽出する。AIによるリライトを通じ、基本的に「～できる」「～できるようになる」というポジティブな状態目標の表現に変換して提示する。
3.  **短期目標の抽出（原因分析の遡及と除外判定）:**
    *   **条件:** ニーズの構造を構成している結果の「○（客観的事実）」を起点とし、「→（順接の矢印）」を逆方向（結果から原因の方向へ）にたどるグラフ走査を実行する。
    *   **抽出ルール:** 「○（原因）→○（結果）」と段階的に細分化されて繋がっている場合、原因分析の最も上流にある「○（原因となる客観的事実）」を短期目標の候補として抽出する。
    *   **例外処理（解決不可能な事実の除外）:** ただし、「転倒」「骨折」など、医学的あるいは物理的にすでに発生して「解消しようがない客観的事実」が最上流に存在する場合、それを短期目標に設定することは不適切である。その場合、システムは順接のパスを1つ下流側へ戻り、次点となる「解決可能な客観的事実」を短期目標の対象としてフォールバック抽出するアルゴリズムを実装する。
4.  **客観的事実が円環（ループ）になっている場合の解釈と抽出（悪循環の断ち切り）:**
    *   **条件:** 「○（客観的事実）」を順接（→）で遡る過程で、パスが循環しループ（閉路）を形成している場合（例：意欲低下 → 活動量低下 → 筋力低下 → さらに意欲低下...）。
    *   **解釈:** これは単一の最上流原因が存在するのではなく、複数の要因が互いに悪化させ合っている「悪循環（Vicious Cycle）」の構造であると解釈する。
    *   **抽出ルール:** グラフ走査アルゴリズムは無限ループを検知した時点で遡及処理を停止する。円環には「最上流」が存在しないため、システムは円環を構成している「○（客観的事実）」のループ群全体を評価対象とする。AI（またはルールの重み付け）を用いて、そのループ群の中から「最も介入が容易な事実」、あるいは「解決可能な事実」を優先的に特定し、その悪循環を断ち切るためのポイントを短期目標の候補として抽出する。

### 10.3 UIへの提示とユーザーフィードバック

上記のアルゴリズムによって抽出された短期・長期目標候補は、LLM（Gemini 3 Proなど）によって自然な「目標の文体」にリライトされた上で、ノンモーダルなUI（ボトムシート等）に表示される。ユーザーは提示された目標候補を確認し、タップ一つでケアプランシステムへ反映、または内容を微修正できる設計とする。

### 10.4 interRAI CAPs（協働的ケア計画）に基づくトリガー論理の統合

関係性ダイアグラムの源流であるインターライ（interRAI）方式のCAPs（Collaborative Action Plans：協働的ケア計画）の概念を、AIの推論モデルに直接組み込み、より専門的で一貫性のあるケアプラン作成を支援する。CAPsはケアプランを単に自動化するものではなく、利用者と専門職が対話を通じて目標達成に向けたステップを決定するための指針（ガイドライン）を提供するものである。

*   **CAPsトリガーマッピング:** キャンバス上に配置された「○（客観的事実）」のノード内容（例：転倒履歴、食事量の減少、排泄の失敗など）を、AIがinterRAIの標準的なCAPs領域（機能的パフォーマンス、認知・精神、社会生活、臨床的問題など）に自動的にマッピングする。
*   **2段階のリスク評価（Trigger Levels）:** CAPsのロジックに倣い、抽出された課題に対して「機能低下のリスクが高い状態（Triggered to prevent decline）」であるか、あるいは急性疾患からの回復期など「改善の可能性が高い状態（Triggered to facilitate improvement）」であるかの2つのトリガーレベルをAIに判定させる。
*   **エビデンスに基づくガイドラインの提示:** 目標の抽出アルゴリズム（10.2）に加えて、該当するCAPsの構造（問題の定義、ケアの全体目標、ケア指針）に基づくアプローチを付与する。これにより、AIは単なる目標の文字列だけでなく、「状態を安定させるための生活環境の整備」や「リハビリによる筋力向上」といった、具体的かつエビデンスに基づいた第2表（ニーズ、長期・短期目標、援助内容）のドラフトを包括的に提案することが可能となる。

## 11. ビジネスモデルおよびライセンスに関する仕様

本アプリケーションを継続可能かつ適切な形でユーザーへ提供するため、以下のビジネスモデルおよび著作者とのライセンス要件を定義する。

### 11.1 収益化・価格設定戦略
医療・介護従事者の個人利用を促しつつ、AI（LLM）のAPI通信による運用コストの赤字（API破産）を回避するため、段階的なサブスクリプションモデルを採用する。

*   **ベーシックプラン（月額500円〜980円程度）:**
    *   個人ユーザーがポケットマネーで導入しやすい価格帯。
    *   機能制限：関係性ダイアグラムの作成・保存は無制限だが、AIによる「ケアプラン目標提案」の実行回数に上限（例：月間50回まで）を設ける。
*   **プロ/アンリミテッドプラン（月額1,980円〜2,980円程度）:**
    *   汎用AI（ChatGPT Plus等の約3,000円）の相場よりわずかに安い価格帯で、専用ツールとしての優位性を提示。
    *   機能制限：AI提案機能を実質無制限で利用可能。ただし、Bot等による不正利用や過剰リクエストを防ぐため、システム内部で「1日あたり50回まで」「1分間に連続3回まで」といったフェアユースポリシー（レートリミット）を必ず実装する。

### 11.2 知的財産（IP）とライセンス契約のガイドライン
関係性ダイアグラムの考案者（吉島豊禄氏および関係性ダイアグラム研究会）のメソッドを商用アプリ化するにあたり、以下の基準をベースに正式な利用許諾・レベニューシェアのスキームを確立する。

*   **利益ベースのロイヤリティ（25%ルール）:**
    本アプリはAIのAPI通信費という重い変動費を伴うため、売上総額ではなく「営業利益（売上からApple/Googleのプラットフォーム手数料やAPI代、サーバー代を差し引いた金額）」をベースとした配分が開発者にとって最もリスクが低い。業界の経験則である「25%ルール」に則り、営業利益の25%をライセンサー（著作者）に支払う形式を第一候補とする。
*   **売上ベースのロイヤリティ（通常実施権）:**
    売上を基準とする場合、一般的なソフトウェアやメソッドの利用における相場である「アプリ総売上の3%〜5%」を目安とする。他者の類似アプリ開発を禁じる独占契約（専用実施権）を結ぶ場合は、10%前後への引き上げを検討する。
*   **レベニューシェア（共同開発・監修）:**
    著作者にアルゴリズムの監修や公式としてのプロモーション協力を依頼する「共同事業」の形を取る場合は、利益の10%〜20%を配分するレベニューシェア契約を締結する。
